# Interactive Sandboxes

Boltdocs features a high-performance, project-agnostic CodeSandbox API. This allows you to create immersive, live-coding experiences that aren't tied to the framework itself—perfect for showcasing design systems, complex libraries, or interactive demos.

## The Core API

The `@boltdocs/client` package handles the heavy lifting of parameter generation and cross-origin communication.

### `openSandbox`

The most direct way to launch a sandbox. It automatically handles URL generation and navigation.

```tsx
import { openSandbox } from 'boltdocs/client';

const handleTryItOut = () => {
  openSandbox({
    title: "Motion Animation Demo",
    template: "node", // Use 'node' for full control or 'vite' for quick starts
    dependencies: {
      "framer-motion": "latest",
      "lucide-react": "latest"
    },
    files: {
      "App.tsx": {
        content: `import { motion } from "framer-motion";
export default function App() {
  return (
    <motion.div animate={{ scale: 1.2 }} transition={{ duration: 0.5 }}>
      Hello Boltdocs!
    </motion.div>
  );
}`
      }
    }
  });
};
```

### `defineSandbox`

If you need to generate parameters for a custom form or a specialized export button, use `defineSandbox`. It returns the encoded parameters and the target URL.

```tsx
import { defineSandbox } from 'boltdocs/client';

const { parameters, url } = defineSandbox({
  files: {
    "index.js": { content: "console.log('Generic demo');" }
  },
  template: "vite"
});
```

## `SandboxOptions` Reference

The configuration object accepted by `openSandbox` and `defineSandbox`:

<Field name="files" type="Record<string, SandboxFile>">
  A map of file paths to their content. Each value is an object with a `content` property (string or object) and an optional `isBinary` boolean flag.
</Field>

<Field name="dependencies" type="Record<string, string>">
  NPM packages to include in the sandbox `package.json`. Keys are package names and values are semver ranges (e.g., `"latest"`, `"^18.2.0"`).
</Field>

<Field name="devDependencies" type="Record<string, string>">
  NPM dev dependencies to include in the sandbox.
</Field>

<Field name="title" type="string">
  The title of the CodeSandbox project, shown in the browser tab.
</Field>

<Field name="description" type="string">
  A short description for the CodeSandbox project metadata.
</Field>

<Field name="template" type="string">
  The CodeSandbox template to use (e.g., `"node"`, `"vite"`, `"static"`).
</Field>

## Advanced Usage: Design System Presets

The real power comes from creating "Presets"—reusable configurations that simulate your actual production environment. 

### Building a Custom Preset

Instead of manually defining files every time, create a helper that injects your global styles and configuration.

```tsx
// src/utils/sandbox.ts
import { openSandbox, SandboxOptions } from 'boltdocs/client';

/**
 * Launches a sandbox pre-configured with our Design System
 */
export function openBrandSandbox(componentCode: string, options: SandboxOptions = {}) {
  openSandbox({
    ...options,
    files: {
      "styles.css": { content: ":root { --brand-primary: #7f13ec; }" },
      "App.tsx": { content: componentCode },
      ...options.files
    },
    dependencies: {
      "my-ui-library": "workspace:*",
      "react": "^18.2.0",
      ...options.dependencies
    }
  });
}
```

## Playground Integration

The built-in `<Playground />` component leverages this API under the hood. You can customize its behavior by passing `sandboxOptions`.

### custom `sandboxOptions` example

```tsx
<Playground 
  code={code}
  sandboxOptions={{
    title: "Brand Component Preview",
    dependencies: { 
      "canvas-confetti": "latest" 
    }
  }}
/>
```

> [!TIP]
> Use `workspace:*` for dependencies if you are building a monorepo documentation site. Boltdocs will handle the resolution logic during the export.

> [!NOTE]
> Boltdocs also supports **StackBlitz**. Check out the [StackBlitz documentation](/docs/integrations/stackblitz) for more information on how to use it as an alternative.

## Why this matters

Generic sandboxes mean your documentation isn't just a manual—it's an IDE for your users. By providing pre-configured environments, you lower the barrier to entry and let them experiment with your code instantly, without leaving the browser.
