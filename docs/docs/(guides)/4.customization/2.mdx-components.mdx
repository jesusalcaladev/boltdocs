# MDX Components

Boltdocs allows you to customize the components used in your MDX files. You can override default HTML elements (like `h1`, `pre`, or `a`) or define custom React components that are globally available in all your Markdown files without needing explicit imports.

## 1. The `mdx-components.tsx` File

To get started, create an `mdx-components.tsx` (or `.ts`, `.js`, `.jsx`) file in your **documentation directory** (e.g., inside `docs/`). 

Boltdocs automatically discovers this file and passes its exports to the MDX provider.

### Basic Setup

The file should have a **default export** that returns an object where keys are the component names or tags you want to use in MDX.

```tsx
// docs/mdx-components.tsx
import React from 'react';

export default {
  // Overriding a standard HTML element
  h1: (props: any) => (
    <h1 style={{ color: '#7f13ec' }} {...props} />
  ),
  
  // Adding a custom component
  MyBadge: ({ children }: any) => (
    <span className="bg-purple-100 text-purple-800 px-2 py-1 rounded">
      {children}
    </span>
  ),
};
```

## 2. Overriding HTML Elements

You can override any standard HTML tag generated by the MDX compiler. This is useful for adding consistent styling or anchors to your headings.

| Tag | Recommended Component | Purpose |
| :--- | :--- | :--- |
| `h1`-`h6` | `Heading` | Customize heading styles or add copyable anchors. |
| `pre` | `CodeBlock` | Replace standard code blocks with custom syntax highlighters. |
| `a` | `Link` | Automatically use `react-router-dom` for internal links. |
| `img` | `Image` | Implement lazy loading or theme-aware image variants. |

### Example: Custom Link Heading

```tsx
export default {
  h2: ({ children, id }: any) => (
    <h2 id={id} className="group flex items-center gap-2">
      {children}
      <a href={`#${id}`} className="opacity-0 group-hover:opacity-100 transition-opacity">
        #
      </a>
    </h2>
  ),
};
```

## 3. Global Custom Components

One of the most powerful features of `mdx-components.tsx` is the ability to define components that can be used anywhere in your documentation without an `import` statement.

### In `mdx-components.tsx`:

```tsx
export default {
  Card: ({ children, title }: any) => (
    <div className="border rounded-lg p-4 shadow-sm bg-white dark:bg-zinc-900">
      <h3 className="font-bold border-b pb-2 mb-2">{title}</h3>
      {children}
    </div>
  ),
};
```

### In any MDX file:

```mdx
# Welcome

This card is available everywhere!

<Card title="Quick Tip">
  You don't need to import components defined in mdx-components.tsx!
</Card>
```

---

## 4. Best Practices

- **Keep it Lightweight**: Since these components are injected into the global MDX provider, keep them focused on layout and presentation.
- **Use TypeScript**: Define prop interfaces for your components to get better IDE support even inside MDX files.
- **Consistent Design**: Use the `mdx-components.tsx` file to enforce your brand's typography and spacing across all document elements.

<Tip>
  Need to use context within your custom components? They are rendered inside the standard `MDXProvider`, so you can access all Boltdocs contexts (like `useConfig` or `useTheme`) normally.
</Tip>
