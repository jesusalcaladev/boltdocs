# Internationalization (i18n)

Boltdocs handles multi-language documentation through a **folder-based convention**. Instead of complex translation files, you simply mirror your docs structure inside locale-named directories. Boltdocs handles routing, language switching, and smart redirects automatically.

---

## Configuration

Enable i18n by adding the `i18n` property to your `boltdocs.config.ts`:

```ts
import { defineConfig } from 'boltdocs'

export default defineConfig({
  i18n: {
    defaultLocale: 'en',
    locales: {
      en: 'English',
      es: 'Español',
      fr: 'Français',
      ar: 'العربية',
    },
    localeConfigs: {
      ar: {
        label: 'العربية',
        direction: 'rtl',
        htmlLang: 'ar',
      },
    },
  },
})
```

<Field name="defaultLocale" type="string" required>
  The default language code. Files in the root `docs/` directory (without a locale subfolder) are treated as this locale. The default locale does **not** get a URL prefix — `/docs/introduction` instead of `/docs/en/introduction`.
</Field>

<Field name="locales" type="Record<string, string>" required>
  A map of locale codes to their default display names. Each code must match a folder name inside `docs/`.
</Field>

<Field name="localeConfigs" type="Record<string, BoltdocsLocaleConfig>">
  Optional per-locale configuration:

  | Property | Type | Description |
  | :--- | :--- | :--- |
  | `label` | `string` | Display name (overrides the value from `locales`) |
  | `direction` | `'ltr' \| 'rtl'` | Text direction for the locale |
  | `htmlLang` | `string` | Value for the HTML `lang` attribute (e.g., `'es-ES'`) |
  | `calendar` | `string` | Calendar system (e.g., `'gregory'`) |
</Field>

## Folder Structure

Place translated files into directories named after their locale codes. The default locale content lives at the root of `docs/`.

<FileTree>
- docs/
  - index.mdx                 # English (default) → /docs
  - getting-started.mdx       # English → /docs/getting-started
  - es/
    - index.mdx               # Spanish → /docs/es
    - getting-started.mdx     # Spanish → /docs/es/getting-started
  - fr/
    - index.mdx               # French → /docs/fr
</FileTree>

<Note>
  The default locale does **not** require its own folder. Files at the root of `docs/` automatically belong to `defaultLocale`.
</Note>

## The Language Switcher

When i18n is configured, Boltdocs automatically injects a **Language Switcher** into the navbar. It works like this:

- **Smart redirection**: When switching from `/docs/getting-started` to Spanish, it navigates to `/docs/es/getting-started` if that page exists.
- **Fallback**: If the target page doesn't exist in the new locale, it redirects to that locale's root page.
- **Persistence**: The selected locale is stored in Zustand (persisted across page refreshes).

## Translating UI Strings

Several configuration fields accept a `Record<string, string>` to provide per-locale translations:

```ts
export default defineConfig({
  theme: {
    // Translated site title
    title: {
      en: 'My Documentation',
      es: 'Mi Documentación',
    },
    // Translated navbar labels
    navbar: [
      {
        label: { en: 'Home', es: 'Inicio' },
        href: '/',
      },
    ],
    // Translated tab text
    tabs: [
      { id: 'guides', text: { en: 'Guides', es: 'Guías' }, icon: 'Book' },
    ],
  },
})
```

## Combining i18n + Versioning

When using both features, the URL structure nests versions **before** locales:

```
/docs/{version}/{locale}/{path}
```

<FileTree>
- docs/
  - v2/
    - index.mdx               # English v2 → /docs/v2
    - es/
      - index.mdx             # Spanish v2 → /docs/v2/es
  - v1/
    - index.mdx               # English v1 → /docs/v1
</FileTree>

## Available Hooks

Boltdocs provides two hooks for working with locales in custom components:

### `useI18n()`

```tsx
import { useI18n } from 'boltdocs/hooks'

function LanguagePicker() {
  const { currentLocale, availableLocales, handleLocaleChange } = useI18n()

  return (
    <select
      value={currentLocale}
      onChange={(e) => handleLocaleChange(e.target.value)}
    >
      {availableLocales.map((locale) => (
        <option key={locale.key} value={locale.value}>
          {locale.label}
        </option>
      ))}
    </select>
  )
}
```

### `getTranslated(value, locale)`

A helper function to resolve a string from a multi-language configuration object. This is useful when manually handling fields that accept both strings and locale-specific objects.

```tsx
import { getTranslated, useI18n } from 'boltdocs/client'

function MyComponent({ titleConfig }) {
  const { currentLocale } = useI18n()
  // titleConfig: { en: 'Hello', es: 'Hola' }
  const title = getTranslated(titleConfig, currentLocale)
  
  return <h1>{title}</h1>
}
```

<Tip>
  The built-in `Link` component from `boltdocs/primitives` already uses `useLocalizedTo` internally — you don't need to call it manually for normal navigation links.
</Tip>
