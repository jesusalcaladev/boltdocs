# Plugin System

Boltdocs features a robust and secure plugin system that allows you to extend the build pipeline, inject Vite configurations, and register global components. Built with a **Safe-by-Design** methodology, every plugin operates within a sandbox, ensuring it only accesses the capabilities you explicitly grant.

---

## Methodology: Secure Sandboxing

Boltdocs plugins are not just standard extensions; they are **Secure Plugins**. This means:

1.  **Explict Permissions**: Plugins must declare which capabilities they need (e.g., `fs:read`, `mdx:remark`).
2.  **Capability Isolation**: Capabilities are only unlocked if the corresponding permission is present. A plugin missing `vite:config` cannot inject Vite plugins.
3.  **Error Isolation**: If a plugin fails during a lifecycle hook, Boltdocs isolates the error, allowing the rest of the build and other plugins to continue.

---

## The `createPlugin` Utility

The recommended way to define a plugin is using the `createPlugin` utility. This provides full TypeScript autocomplete and ensures your plugin conforms to the `SecureBoltdocsPlugin` interface.

```ts
import { createPlugin } from 'boltdocs'

export default createPlugin({
  name: 'my-plugin',
  permissions: ['mdx:remark'],
  remarkPlugins: [/* ... */],
})
```

## Permissions API

Permissions are the foundation of the Boltdocs sandbox. Here are the available values:

| Permission | Capability Unlocked |
| :--- | :--- |
| `fs:read` | Grants permission to read from the local filesystem. |
| `fs:write` | Grants permission to write to the local filesystem. |
| `vite:config` | Allows injecting plugins into the **Vite** build process. |
| `mdx:remark` | Allows adding **Remark** plugins to the MDX pipeline. |
| `mdx:rehype` | Allows adding **Rehype** plugins to the MDX pipeline. |
| `components` | Allows registering **Global Components** for MDX. |
| `hooks:build` | Enables access to build-time lifecycle hooks (`beforeBuild`, etc.). |
| `hooks:dev` | Enables access to dev-server lifecycle hooks (`beforeDev`, etc.). |

## The Plugin Interface

<Field name="name" type="string" required>
  A unique identifier (e.g., `boltdocs-plugin-mermaid`).
</Field>

<Field name="permissions" type="PluginPermission[]">
  A list of required capabilities. **Mandatory** if you use remark, rehype, vite, or hooks.
</Field>

<Field name="enforce" type='"pre" | "post"'>
  Execution order. `"pre"` runs before core processing, `"post"` runs after.
</Field>

<Field name="hooks" type="PluginLifecycleHooks">
  Implementation of build or dev lifecycle ganchos.
</Field>

## Lifecycle Hooks

Hooks allow you to execute code at specific points in the Boltdocs lifecycle. Every hook receives a `PluginContext`.

| Hook | Permission Required | Description |
| :--- | :--- | :--- |
| `beforeBuild` | `hooks:build` | Runs before the documentation build process starts. |
| `afterBuild` | `hooks:build` | Runs after the build completes successfully. |
| `beforeDev` | `hooks:dev` | Runs before the development server starts. |
| `afterDev` | `hooks:dev` | Runs once the development server is up and running. |
| `configResolved` | *None* | Runs after Boltdocs has fully resolved the configuration. |
| `buildEnd` | `hooks:build` | Runs when the build process is closing/tearing down. |

### The `PluginContext`

Every hook is injected with a context object containing:

- `config`: Read-only access to the resolved Boltdocs configuration.
- `logger`: A namespaced logger for consistent console output.
- `store`: A shared key-value store for state sharing between plugins.
- `meta`: Information about the current plugin (name, version).

---

## Shared State & Dependency Injection

Plugins can share information or provide services to other plugins using the `store`.

```ts
// Provider Plugin
hooks: {
  beforeBuild: (ctx) => {
    ctx.store.set('my-plugin', 'service-ready', true)
  }
}

// Consumer Plugin
hooks: {
  beforeBuild: (ctx) => {
    const isReady = ctx.store.get('my-plugin', 'service-ready')
    if (isReady) ctx.logger.info('External service is ready!')
  }
}
```

## Examples

### 1. Basic: MDX Transformer
A simple plugin that adds a Remark transformation.

```ts
import { createPlugin } from 'boltdocs'
import { visit } from 'unist-util-visit'

export default function alertPlugin() {
  return createPlugin({
    name: 'boltdocs-plugin-alerts',
    permissions: ['mdx:remark'],
    remarkPlugins: [
      () => (tree) => {
        visit(tree, 'text', (node) => {
          if (node.value.includes('!!')) {
            node.value = node.value.replace('!!', '⚠️')
          }
        })
      }
    ]
  })
}
```

### 2. Advanced: Lifecycle Integration
A plugin that monitors build time and logs it to a file.

```ts
import { createPlugin } from 'boltdocs'
import fs from 'fs'

export default function auditPlugin() {
  let startTime: number

  return createPlugin({
    name: 'boltdocs-plugin-audit',
    permissions: ['hooks:build', 'fs:write'],
    hooks: {
      beforeBuild: (ctx) => {
        startTime = Date.now()
        ctx.logger.info('Build audit started...')
      },
      afterBuild: (ctx) => {
        const duration = Date.now() - startTime
        const log = `Build completed in ${duration}ms\n`
        fs.appendFileSync('./build-audit.log', log)
        ctx.logger.info(`Audit complete: ${duration}ms`)
      }
    }
  })
}
```
